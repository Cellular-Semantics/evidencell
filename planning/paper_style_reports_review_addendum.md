## 2. Methods provenance — what's already in the schema

| Class | Slot(s) | Provenance covered |
|---|---|---|
| `LiteratureEvidence` | `reference`, `snippet`, `study_type` | Source paper identity |
| `AtlasMetadataEvidence` | `atlas`, `cell_set_accession`, `metadata_url` | Atlas source |
| `AnnotationTransferEvidence` | `method`, `tool_version`, `code_reference`, `source_dataset_accession`, `source_species`, `target_species`, `bootstrap_threshold`, `metrics_by_level` | **Strongest existing methods provenance.** `code_reference` already supports a notebook/script URL. |
| `BulkCorrelationEvidence` | `run_ref`, `contrast_ref`, `target_accession`, `statistics` | Thin pointer; details flow via `run_ref` |
| `BulkDataset` | `source_pmid`, `geo_accession`, `authors`, `year`, `title`, `data_files[]`, `pools[]` | Source publication + raw data |
| `BulkDataFile` | `relpath`, `url`, `sha256`, `keyed_by`, `normalisation` | **Raw data version pinned by SHA-256** |
| `AtlasReference` | `taxonomy_id`, `pseudobulk_source`, `sha256` | **Atlas data version pinned by SHA-256** |
| `CorrelationMethod` | `statistic_kind`, `parameters` | Statistic family + free-form params |
| `CorrelationRun` | `dataset_ref`, `atlas`, `method`, `contrasts`, `script`, `output`, `caveats`, `metadata` | Run-level reproducibility receipt |
| `ScriptReference` | `relpath`, `python_version`, `packages` | Path + Python version + dependency list |

What's already auditable today: a reader looking at a `BulkCorrelationEvidence` item on a mapping edge can:
1. Follow `run_ref` to the run manifest
2. Confirm input data SHAs against the dataset YAML
3. See the script path, Python version, and dependencies
4. See the atlas pseudobulk SHA
5. Read off the exact statistic and parameters

That's already strong.

## 3. Identified gaps and proposed schema tweaks

### Gap 1 — Code provenance is local-path-only
`ScriptReference.relpath` is repo-relative. Without a repo URL + commit hash, a reader can't form a stable permalink to the exact code that was run. `AnnotationTransferEvidence.code_reference` (free string URL) handles this for AT but the pattern isn't reusable for `CorrelationRun.script` etc.

### Gap 2 — No render-time provenance
Reports don't carry the evidencell git commit at gen time. Two reports for the same node generated by different versions of `render.py` produce slightly different output; a reader has no way to tell which version produced any given report.

### Gap 3 — Figure-to-report sync (Q3)
Without a sync mechanism, a report can reference `figure_a.png` and the file on disk could be from a different run with different data. The reader sees stale visuals and the report says nothing about it.

### Gap 4 — Methods section needs to surface AT data sources cleanly
AT evidence already has `method`, `tool_version`, `code_reference`, `source_dataset_accession` — fine for citation. But the Methods subsection for AT should also note the F1 distribution across other levels (already in `metrics_by_level`), the bootstrap threshold, and the cell counts (`n_cells_total`, `n_cells_after_filter`). All present in schema; no schema change.

### Proposed schema tweaks (4 small additions, 0 breaking changes)

**(a) `ScriptReference.git_repo_url` + `git_commit`** *(both optional)*
```yaml
git_repo_url:
  description: Repository URL where the script lives (canonical).
  pattern: "^https://(github|gitlab)\\.com/.+$"
git_commit:
  description: >
    Commit SHA or tag pinning the exact version of the script that was run.
    Combined with git_repo_url + relpath, gives a stable permalink for audit.
```

**(b) `CorrelationRun.code_version`** *(optional)*
A simple per-run string for the codebase version, e.g. `"v0.6.0"` or `"a279270"` (commit hash). Can be omitted if the script's `git_commit` covers it.

**(c) `Figure` class + `MappingEdge.figures` slot** *(optional, new — used by figure rendering)*
```yaml
Figure:
  description: A programmatically generated figure attached to a mapping edge or report.
  attributes:
    relpath:
      required: true
      description: Path under reports/{region}/figures/, content-hashed.
    figure_type:
      description: e.g. "bulk_correlation_top_n", "expression_heatmap", "delta_distribution"
    inputs_sha:
      description: SHA-8 of the input data the figure was rendered from.
    rendered_at:
      range: datetime
    code_version:
      description: evidencell commit or version when the figure was rendered.
    caption:
      description: One-line figure caption for the report.
```

This class can also live in a sidecar `.meta.yaml` next to each figure rather than in the KB graph — depends on whether we want figures discoverable from the graph. **Recommendation: sidecar `.meta.yaml` for now**, no schema class. The graph already over-references things; figures are render-time artefacts and don't need first-class graph status.

So the actual schema delta is **just (a) + (b)** — two small additions on `ScriptReference` and `CorrelationRun`.

## 4. Figure-to-report sync mechanism

The user's concern (Q3): figures could go out of sync with the report that references them.

### Approach: content-hashed filenames + sidecar metadata

```
reports/{region}/figures/{node_id}_{contrast_id}_{inputs_sha8}.png
reports/{region}/figures/{node_id}_{contrast_id}_{inputs_sha8}.meta.yaml
```

- `inputs_sha8`: first 8 chars of SHA-256 over the figure's input data (the contrast's ranked TSV bytes, plus any rendering parameters in canonical form).
- Each figure is accompanied by a sidecar YAML with: render timestamp, evidencell commit, contrast id, input file SHA, rendering params, caption.

### Why this gives sync

- Report references `![caption](figures/foo_a3f72b8c.png)`. The hash in the filename pins what the report saw.
- If the underlying TSV changes, regenerating the figure produces a *different* hash → different filename. The old reference in the report is now a broken link, which is loud and fixable.
- The sidecar YAML lets a reader confirm: "this figure was generated by evidencell at commit X from data SHA Y on date Z". Equivalent to a paper's supplementary figure metadata.
- Old figures can be GC'd if not referenced anywhere; or kept as archival.

### Why not git-LFS or large binary commits

PNGs are small (10-100 KB each); committing them works fine. Hash-versioning keeps history tidy without git-LFS complexity.

## 5. Methods section content (with reproducibility focus)

Updated content list for the Methods `<details>` fold (reflecting Q1 = fold, Q2 = defer find-candidates persistence, Q5 = generic warning here):

**Inside the fold, in this order:**

1. **Classical type definition** — one paragraph from `classical_nodes[0]` (markers, NT, anatomy, supporting publications). Cites references inline.
2. **Atlas mapping query** — static text: "Candidate atlas clusters were retrieved from {taxonomy_id} at ranks 0 (cluster) and 1 (supertype) using metadata-based scoring (region match, NT type, defining markers). The full scoring rules are documented in `workflows/map-cell-type.md`." Specific run params deferred per Q2.
3. **Property alignment** — one paragraph noting that each defining property was compared between classical and atlas representations using the `property_comparisons` schema; alignments are CONSISTENT / APPROXIMATE / DISCORDANT / NOT_ASSESSED. Numbers come from atlas precomputed expression and the bulk RNA-seq references.
4. **Annotation transfer** *(if AT items present)* — table or paragraph with: dataset accession, method, tool_version, F1 score(s), bootstrap threshold, code_reference link.
5. **Bulk transcriptomic correlation** *(if BulkCorrelation items present)* — table with: BulkDataset source publication ([n] miniref), GEO accession, technique, statistic_kind, parameters, contrast pair, n_pools, n_genes_shared, top-N hits subtable. **Each row has a "data + code" cell** showing the dataset SHA + script git_commit (when available) so the reader can audit.
6. **Reproducibility footer** — auto-generated by the renderer: evidencell commit at gen time, facts JSON timestamp, KB graph file relative path. Brief, single line.
7. **Framework limitations** *(generic warning)* — one paragraph: "evidencell mappings are evidence summaries, not authoritative cell-type identity claims. Mapping confidence is qualitative; absence of an evidence type does not imply absence of the underlying biology. The KB and references are versioned; specific limitations of any individual mapping are noted in the Discussion."
