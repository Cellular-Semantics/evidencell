# Taxonomy Update Architecture — Design Spike

**Status:** Implemented (add-expression + reingest); update-anatomy, update-names deferred
**Date:** 2026-04-23
**Context:** Phase 3 of the male_female_ratio plan. Addresses how taxonomy
reference YAML is maintained after initial ingest, without data loss on
re-ingest and without uncontrolled overlay file proliferation.

---

## Problem

Taxonomy reference YAML (`kb/taxonomy/{id}/*.yaml`) is currently generated by
a flush-and-replace ingest pipeline. Post-ingest enrichments (e.g.
`precomputed_expression` blocks, curated anatomy corrections) have nowhere
safe to live — they either go on mapping graph stubs (wrong place) or get
clobbered on re-ingest.

We need a managed update layer that:
- Supports selective field-level updates (not just full re-ingest)
- Preserves enrichments across re-ingests
- Validates all writes against the LinkML schema
- Produces clean git diffs (no text patching, no YAML surgery)

---

## Change patterns (from experience)

| Change type | Frequency | Scope | Example |
|---|---|---|---|
| Anatomy remapping | 2-3x post-release | Field update on many nodes | CCF v3 → v4 region reassignment |
| Name changes | Occasional | Field update on many nodes | Cluster label standardisation |
| New markers | Moderate | Append to list field on specific nodes | Precomputed stats query results |
| Expression profiles | Per-mapping | New block on specific nodes | precomputed_expression from HDF5 |
| Full taxonomy version | Rare | Complete replacement | CCN20230722 → CCN2024xxxx |

Key observation: most updates are **field-level edits to existing nodes**,
not additions of new nodes or new property types. An overlay/extension file
system is the wrong abstraction — these are mutations, not layers.

## Proposed approach: typed Python model with operation-based updates

### Core architecture

```
schema/celltype_mapping.yaml
        |
        v  (gen-pydantic)
src/evidencell/model.py          # Generated Pydantic CellTypeNode classes
        |
        v  (load)
Taxonomy YAML  ←→  Python objects  ←→  Updated YAML
                        ^
                        |
                   Update operations
```

1. **Generate Pydantic models** from `schema/celltype_mapping.yaml`
   → `src/evidencell/model.py`. Run on schema changes, commit the output.

2. **Load taxonomy YAML** into typed `CellTypeNode` objects via
   `linkml-runtime` `yaml_loader` (already a dependency).

3. **Apply update operations** — each operation is a Python function that
   loads the model, mutates specific fields, and writes back.

4. **Serialize back** to YAML via `yaml_dumper` with consistent formatting.
   Git tracks the diff.

### Update operations (not exhaustive)

Each operation is a thin Python function or CLI subcommand. The operation
knows which fields it owns and only touches those.

#### `update-anatomy` — re-map anatomical locations

```
just update-anatomy {taxonomy_id} [--source kg|ccf-json|manual]
```

- Loads all nodes for taxonomy
- Replaces `anatomical_location` entries from new source
- Preserves all other fields (markers, expression, etc.)
- Source options:
  - `kg`: re-query the knowledge graph for current anatomy
  - `ccf-json`: parse a CCF region assignment file
  - `manual`: read a TSV of `accession → [region_id, label, count]`

#### `update-names` — apply name/label changes

```
just update-names {taxonomy_id} [--source kg|tsv]
```

- Updates `name` field on matched nodes
- Optionally updates `label` in parent_hierarchy references (cascade)

#### `add-expression` — add precomputed expression profiles

```
just add-expression {taxonomy_id} {stats_h5} {gene_list} [--accessions ...]
```

- Queries HDF5 for specified genes across specified (or all) nodes
- Writes/updates `precomputed_expression` block
- Idempotent — re-running with same genes overwrites cleanly

#### `reingest` — full re-ingest with enrichment preservation

```
just reingest {taxonomy_id} {source_json} [--preserve-fields expression,precomputed_expression]
```

- Runs standard ingest to a temp location
- Loads both old (enriched) and new (fresh) models
- For each node (matched by accession):
  - Ingest-owned fields: take from new
  - Preserved fields: keep from old
- Nodes in new but not old: add as-is
- Nodes in old but not new: flag for review (don't silently drop)

#### `diff-from-kg` — compare current YAML against fresh KG export

```
just diff-from-kg {taxonomy_id}
```

- Fetches fresh data from KG (or reads a new export JSON)
- Loads current YAML into model
- Compares field by field, reports differences
- Does NOT write — human reviews diff and decides which update operations to run

### Field ownership convention

Rather than per-field provenance tags, use a simple convention:

```python
# In each update operation, declare what you touch:
INGEST_FIELDS = {"name", "markers", "nt_type", "anatomical_location",
                 "parent_hierarchy", "cl_mapping", "neighborhood",
                 "male_female_ratio", "taxonomy_rank"}

EXPRESSION_FIELDS = {"precomputed_expression"}

# reingest() preserves everything not in INGEST_FIELDS
```

This is explicit, auditable, and doesn't require schema annotations.
If a new enrichment type is added, its update operation declares its fields.

## Key design decisions

### Single file per level (not overlays)

Rationale:
- Most changes are field-level mutations, not additive layers
- Single file = single source of truth, clean git diffs
- No merge logic, no conflict resolution, no file discovery
- The Python model validates the whole file on every write

### Pydantic over raw dict manipulation

Rationale:
- Schema validation on every write (catches bad edits immediately)
- Type safety in update operations
- Round-trip fidelity (Pydantic → YAML is deterministic)
- Already have `gen-pydantic` and `linkml-runtime` as dependencies

### Operations over general-purpose editor

Rationale:
- Each operation is bounded and testable
- Operations declare field ownership explicitly
- Easier to audit ("what changed the anatomy?" → `update-anatomy` commit)
- Prevents ad hoc YAML surgery that bypasses validation

## YAML serialization concerns

`linkml-runtime` `yaml_dumper` may produce different formatting than our
current `yaml.dump()` output (key ordering, quoting, flow style). Options:

1. Accept a one-time reformatting diff on first round-trip (then stable)
2. Use a custom YAML dumper that matches current conventions
3. Use `ruamel.yaml` for round-trip-preserving edits

Option 1 is simplest. The reformatting diff is a one-time cost and
subsequent diffs are clean. Worth testing with a sample file first.

## Open questions

- **ruamel.yaml vs linkml yaml_dumper**: Which gives better round-trip
  fidelity? Need to test with a real taxonomy file.
- **Validation scope**: Should update operations validate only the mutated
  nodes, or the entire file? Entire file is safer but slower for 5k+ nodes.
- **Node deletion handling**: When a re-ingest has fewer nodes than current
  YAML, what happens? Flag for review, archive, or silently keep stale nodes?
- **Cross-node consistency**: Some updates cascade (e.g. a name change
  should update parent_hierarchy references in child nodes). How much
  cascade logic do operations need?
- **CLI vs Python API**: Are update operations only CLI subcommands, or do
  orchestrators call them as Python functions? Probably both — CLI for human
  use, Python API for orchestrator integration.
